Process pending PR review drafts generated by `review-prs` (headless batch script), letting the human select which of the remaining low-confidence comments to post.

$ARGUMENTS

The argument can be:
- Empty — process every `*.review.md` file under `/tmp/claude-reviews/`
- A partial filename or repo/PR reference (e.g. `observ`, `observ-pr-12215`, `observ 12215`) — process only matching draft file(s)

## Draft File Structure

Each draft was already run through the confidence pipeline (`pr_confidence_review.py`) before you see it. It has these sections in order:
1. `PR Summary` / `Risk Summary` / `Candidate Review Comments` — the full human-readable draft (context only, not actionable here).
2. `## ✅ Auto-Posted (confidence ≥ 80)` — findings that were already independently verified (via codex adversarial check) and posted to GitHub automatically. **These are done — do not re-ask about them, do not re-post them.** Just note them in your summary for visibility. If this section says posting *failed*, treat those findings as if they were folded into the needs-review list instead (they get a manual decision).
3. `## 🔎 Needs Your Review (confidence < 80)` — the only section requiring a human decision. Each item shows its confidence score and the verification reasoning (why it didn't clear the auto-post bar) alongside the normal `type` / `severity` / `location` / `comment` / `why` / `suggested_fix` fields.

If a draft has neither section (older-format file, or the confidence pipeline could not parse findings — it will say so explicitly with a `⚠️` note), fall back to treating the whole `Candidate Review Comments` list as needs-review.

## Workflow

1. **List pending drafts**: `ls -t /tmp/claude-reviews/*.review.md 2>/dev/null`
   - If none exist, say so and stop (suggest running `review-prs <repo> <pr...>` first).
   - Filter to files matching `$ARGUMENTS` if given.
   - Show the list (repo, PR number, generated time parsed from filename/header) before processing any of them.

2. **Process one draft at a time** (do not batch multiple PRs' comment selection into one prompt — keep each PR's decision separate and clear):
   - Read the draft file in full.
   - If the file looks empty, truncated, or errored (e.g. contains an error trace instead of a review), flag it and skip — don't guess at malformed content.
   - Briefly note what was already auto-posted (count + one-line list), if anything.
   - Display a compact summary table of the **Needs Your Review** items only: `#`, confidence, severity, type, location, one-line gist.
   - If Needs Your Review is empty (everything auto-posted or nothing was found), skip straight to cleanup for this draft — don't ask a pointless question.
   - Otherwise ask: "Which comments do you want to post for `<repo>#<PR>`? Enter numbers (e.g. `1 3 4`), `all`, or `none`."

3. **Confirm before posting**: reprint the selected comments in full, ask for explicit confirmation.

4. **Post on confirmation** — follow the `pr-review-post` skill exactly (same `gh api` mechanics, same body formatting, `event: COMMENT` only, never auto-approve).

5. **After posting (or on `none`)**: delete the processed draft file (`rm`) and, if its worktree directory `/tmp/claude-reviews/<repo>-pr-<N>/` still exists (e.g. the batch run was killed mid-way), clean it up too.

6. **Move to the next pending draft** and repeat until all matching drafts are processed. End with a one-line summary: how many PRs processed, how many comments were already auto-posted, how many were posted from manual selection just now.

## Constraints
- Never post without explicit human confirmation of the exact comment list, per PR.
- Never re-ask about or re-post items already listed under Auto-Posted.
- Never merge comments from two different PRs into one post.
- If a draft's repo isn't resolvable (missing/renamed local checkout), ask the user for the `owner/repo` instead of guessing.
