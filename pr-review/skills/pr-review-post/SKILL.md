# PR Review Post Skill

## Purpose
Take a confirmed list of review comments and post them to GitHub as a single PR review with inline comments using `gh api`.

---

## Prerequisites

Before posting, verify:
```bash
gh auth status
```
If not authenticated, stop and ask the user to run `gh auth login` first.

---

## Required Inputs

The user must provide:
1. **PR reference** — any of these formats:
   - PR number only: `42` (uses current repo directory)
   - Full URL: `https://github.com/owner/repo/pull/42`
   - Owner/repo + number: `acme/api#42`
2. **Confirmed comments** — the subset of candidate comments the human has decided to post

---

## Step 1: Resolve owner, repo, and PR number

**If only a PR number is given** (e.g. `42`):
```bash
gh repo view --json owner,name --jq '[.owner.login, .name] | join("/")'
```
Use the output as `owner/repo`, and the given number as `pr_number`.

**If a full URL is given** (e.g. `https://github.com/acme/api/pull/42`):
Extract `OWNER`, `REPO`, `PR_NUMBER` directly from the URL path.

**If `owner/repo#number` format is given** (e.g. `acme/api#42`):
Split on `#` to get `owner/repo` and `pr_number`.

---

## Step 2: Resolve the head commit SHA

The review API anchors inline comments with `line` + `side` against a commit, so
fetch the PR's head SHA:
```bash
gh api repos/{owner}/{repo}/pulls/{pr_number} --jq '.head.sha'
```
Save it as `commit_id`. (We do not pre-parse the diff to validate line numbers —
if a line isn't in the diff the API returns 422, which the Error Handling table
below falls back to the top-level body.)

---

## Step 3: Build the JSON payload

For each confirmed comment, construct an entry. Write the `body` in a natural
reviewer voice — see the **Comment Voice** section below. Do **not** emit the
`[severity · type]` tag.

```json
{
  "path": "app/services/user.py",
  "line": 128,
  "side": "RIGHT",
  "body": "This early return skips validation when `items` is empty, so downstream code can get unvalidated input. Worth validating before the early return?"
}
```

**Body formatting rules:**
- One or two sentences, plain prose. No `[severity · type]` prefix, no bold
  labels, no emoji.
- Match the severity through wording, not a tag (see **Comment Voice**):
  - `must` → state the concrete failure directly (assertive, but not commanding).
  - `should` → raise it as a concern or a question.
  - `nit` → prefix with `nit: ` (the one marker we keep — it is a standard
    human reviewer convention).
- Fold any suggested fix into the prose (e.g. "…maybe validate first?") rather
  than a separate `Suggested:` line.
- Preserve the **substance** of the confirmed comment — you may rephrase for
  tone, but never change what the finding claims or drop its point.

**Handling unknown locations:**
- If `location` is `unknown` or has no line number, collect these into the top-level review `body` instead of inline comments.

**Line number extraction:**
- If location is `app/services/user.py:128`, use line `128`, path `app/services/user.py`
- If location is a range like `app/services/user.py:128-135`, use the last line `135`

---

## Step 4: Write the payload to a temp file

**Build the top-level `body` first.** This is where the blocker signal lives, so
the author sees it at a glance:

- If there are any `must`-severity findings, open with a short "before merge"
  list of just those, in natural prose. Keep it to the must-fixes; do not repeat
  every inline comment.
- If there are no `must` findings, use a one-line neutral summary (e.g. "Looks
  reasonable overall — a few smaller suggestions inline.").
- Append any unknown-location comments after the summary.
- Write it in the same natural voice as the inline comments. No `[severity]`
  tags, no emoji, no "generated by" footer.

Example top-level body:

```
大致看過，邏輯沒問題，兩個地方想在 merge 前確認一下：
- batch.py:84 — 空 payload 會跳過驗證
- deployment.yaml:47 — env var 名稱和程式讀的對不上

其他都是小建議，寫在 inline。
```

Then assemble the payload:

Use a PR-scoped temp path (`/tmp/pr_review_payload_<pr_number>.json`) so
concurrent reviews of different PRs don't clobber each other:

```bash
cat > /tmp/pr_review_payload_<pr_number>.json << 'EOF'
{
  "commit_id": "<HEAD_SHA>",
  "body": "<top-level summary: must-fix list or neutral one-liner>",
  "event": "COMMENT",
  "comments": [
    {
      "path": "app/services/user.py",
      "line": 128,
      "side": "RIGHT",
      "body": "This early return skips validation when `items` is empty..."
    }
  ]
}
EOF
```

Set `event` to `COMMENT` to post without approving or requesting changes.

---

## Step 5: Post the review

```bash
gh api repos/{owner}/{repo}/pulls/{pr_number}/reviews \
  -X POST \
  --input /tmp/pr_review_payload_<pr_number>.json
```

---

## Step 6: Confirm success

Check the response for `"state": "PENDING"` or `"state": "COMMENTED"`.

Print a summary:
- How many inline comments were posted
- How many (if any) were posted as top-level body text due to missing line info
- Link to the PR review: `https://github.com/{owner}/{repo}/pull/{pr_number}`

Clean up: `rm /tmp/pr_review_payload_<pr_number>.json`

---

## Error Handling

| Error | Action |
|-------|--------|
| `gh auth status` fails | Stop. Ask user to authenticate with `gh auth login` |
| Line not found in diff | Move comment to top-level `body` with its location noted |
| API 422 Unprocessable | The line number may not exist in this commit's diff — fall back to top-level body |
| API 404 | Wrong owner/repo/PR number — ask user to verify the PR URL |

---

## Comment Voice (style guide)

The human has already reviewed and approved the substance of every comment, so
these post in their voice — they should read like notes a human reviewer left,
not machine output. Rephrase for tone; never change what a finding claims.

Rules:
- No `[severity · type]` tags, no bold labels, no emoji, no "generated by" footer.
- First person is fine. When unsure, ask ("is this intended?") instead of asserting.
- Vary phrasing between comments — do not stamp them from one template.
- Keep each comment to one or two sentences.
- Only `nit:` survives as a prefix (standard reviewer shorthand for trivia).
- Severity is carried by wording inline, and by the must-fix list in the
  top-level body — not by per-comment tags.

Before → after:

| Old (machine) | New (human) |
|---|---|
| `**[must · logic]** This early return bypasses validation on empty payload, risking unclear downstream failures.` | `This early return skips validation when items is empty, so downstream code can get unvalidated input — needs handling.` |
| `**[should · logic]** ...` | `I'm a little worried about the boundary case here — if X happens, could we hit Y?` |
| `**[nit · naming]** Rename data to user_rows.` | `nit: data is a bit generic here — user_rows?` |
| `**Suggested**: Validate before the early-return path.` | `…maybe validate before the early return?` |

---

## Important Constraints
- Never post without explicit human confirmation of the comment list
- Always use `event: COMMENT` — never `APPROVE` or `REQUEST_CHANGES` unless the user explicitly asks
- You may rephrase confirmed comments for tone per the **Comment Voice** guide,
  but never change the substance of a finding, drop its point, or add new claims
  the human did not confirm
